# Architecture

`n8n-proxy-auth` ships one CommonJS external hook for upstream `n8nio/n8n`. It does not ship n8n
itself. Operators pin official n8n and this hook artifact independently.

## Goal

The proxy is the identity layer. n8n receives a signed assertion, the hook verifies it, and n8n gets
a native session cookie for the verified user.

Non-goals:

- replacing n8n Enterprise SSO feature-for-feature;
- running an OAuth/OIDC login flow in this project;
- making the first proxy-authenticated user the n8n owner;
- maintaining an n8n-side allow-list or group-to-role sync.

## Request Flow

1. The browser authenticates to an identity-aware proxy.
2. The proxy forwards the request to n8n with a signed identity JWT in a configured header.
3. The hook middleware runs after `cookieParser` and before n8n's per-route auth middleware.
4. Layer 1 verifies the JWT: signature, issuer allow-list, audience, expiry, and pinned algorithm.
5. Layer 2 reconciles any existing `n8n-auth` cookie with the verified identity.
6. If needed, Layer 2 finds or provisions the n8n user and calls n8n's `AuthService.issueCookie`.
7. The request continues into normal n8n route auth with a native `n8n-auth` cookie.

No verified header means no proxy-auth login. The request falls through to n8n's normal unauthenticated
behavior.

## Components

`src/verify.ts` is the pure JWT verifier. It owns the closed `TrustedIssuer[]` registry, remote JWKS
fetching, algorithm pinning, audience checks, header token selection, and claim mapping. It has no
n8n dependency and is covered by unit tests.

`src/jwt-assertion.ts` is a provider-neutral convenience wrapper over the generic verifier.

`src/hook.ts` is the n8n external-hook entrypoint. It exports the `n8n.ready` hook shape that n8n
loads from `EXTERNAL_HOOK_FILES`.

`src/proxy-auth.ts` is the n8n-coupled adapter. It parses env config, splices middleware into the
Express stack, resolves n8n internals, reconciles cookies, provisions users, and issues cookies.

`dist/hook.cjs` is generated by `tsup` and gitignored. Releases publish it two ways:

- `n8n-proxy-auth-hook-vX.Y.Z.tar.gz`, for direct download into a filesystem n8n can read;
- `ghcr.io/<owner>/n8n-proxy-auth-hook:vX.Y.Z`, a one-shot OCI image that copies `hook.cjs` into a
  mounted output directory.

## n8n Internal Couplings

This project intentionally touches unstable n8n internals. The e2e gate is the safety contract for
those couplings.

| Coupling                                                   | Why it exists                                                               | Guardrail                                   |
| ---------------------------------------------------------- | --------------------------------------------------------------------------- | ------------------------------------------- |
| Express router stack splice after `cookieParser`           | `n8n.ready` runs after route auth is mounted; a plain `app.use` is too late | Real-image e2e proves header auth works     |
| `EXTERNAL_HOOK_FILES` and `n8n.ready`                      | Supported hook loading surface                                              | Boot/readiness and installed-probe checks   |
| Anchored `createRequire` into n8n's install tree           | The hook lives outside n8n's package tree                                   | Build output and e2e run inside n8n         |
| `@n8n/di`, `AuthService`, `UserRepository`, role constants | Use n8n's own cookie and user semantics                                     | E2E covers issue-cookie, provision, reuse   |
| Owner setup remains outside the hook                       | Only n8n owner setup flips the owner-ready state                            | E2E seeds owner before proxy-auth scenarios |

If an upstream n8n bump breaks one of these, Renovate should leave a red compatibility-test PR
rather than changing the hook artifact silently.

## Security Invariants

- Verify a JWT signature against configured JWKS; never trust a raw email/header value.
- Match `iss` against a closed allow-list; never derive a JWKS URL from token-controlled data.
- Require a non-empty audience on the deploy path.
- Pin algorithms. Symmetric `HS*` algorithms are rejected for JWKS issuers.
- Key n8n accounts by normalized verified email.
- Reconcile existing cookies against the verified proxy identity. A valid cookie for user A plus a
  valid header for user B re-issues for user B.
- Fail safe. Config/install errors disable proxy auth; per-request errors continue unauthenticated.
- Provisioned users are `global:member`. The proxy policy is the access-control gate.
- Keep n8n reachable only through the proxy.

## Pomerium Notes

The tested real-Pomerium stack uses Pomerium `v0.30.4` and the default `jwt_issuer_format:
IssuerHostOnly`. In that shape:

```text
alg = ES256
iss = n8n.pomerium.localhost
aud = n8n.pomerium.localhost
JWKS = https://<route-host>/.well-known/pomerium/jwks.json
```

Those values are route-host literals, with no scheme and no trailing slash. Future deployments should
still confirm against a live `/.pomerium/jwt` token, because the verifier exact-matches `iss`.

Do not configure route-level removal for `x-pomerium-jwt-assertion`; it can strip Pomerium's own
injected assertion before the request reaches n8n.
